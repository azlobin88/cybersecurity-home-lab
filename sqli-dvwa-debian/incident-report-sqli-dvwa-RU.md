# Отчёт об инциденте: попытка эксфильтрации через SQL-инъекцию (enumeration) на DVWA

Фреймворк: CERT Société Générale IRM-3

---

## 1. Время активности

19 июля 2026, 01:00:16 – 01:00:48 (UTC+3). Зафиксировано пять вредоносных HTTP GET-запросов в течение 32 секунд, все с одного исходного IP-адреса.

---

## 2. Затронутые системы

| Поле | Значение |
|---|---|
| Целевой хост | Debian victim VM — 192.168.1.196 |
| Затронутое приложение | DVWA (Damn Vulnerable Web Application), эндпоинт `/dvwa/vulnerabilities/sqli/` |
| Исходный IP | 192.168.1.133 |
| Веб-сервер | Apache2, источник лога: `/var/log/apache2/access.log` |
| Платформа детектирования | Wazuh manager — 192.168.1.153 |
| Агент Wazuh | agent.id 001 (debian) |

---

## 3. Краткое описание

Хост-источник (192.168.1.133) отправил последовательность из пяти HTTP GET-запросов на страницу SQL Injection в DVWA, каждый из которых содержал UNION-based SQL-инъекцию через параметр `id`. Последовательность payload-ов соответствует классическому паттерну разведки базы данных (enumeration):

1. `1' UNION SELECT null,null -- -` — определение количества колонок
2. `1' UNION SELECT user,password FROM users -- -` — попытка извлечения учётных данных
3. `1' UNION SELECT table_name,null FROM information_schema.tables -- -` — перечисление таблиц
4. `1' UNION SELECT column_name,null FROM information_schema.columns -- -` — перечисление колонок
5. `1' UNION SELECT database(),null -- -` — определение имени базы данных

Все пять запросов вернули HTTP 200 (успешное выполнение, данные возвращены атакующему), что подтверждает: инъекция не была заблокирована или санитизирована приложением.

---

## 4. Логика детектирования

Детектирование реализовано как двухуровневая цепочка правил Wazuh, по аналогии с ранее реализованным SSH brute-force сценарием (атомарная сигнатура → корреляция по частоте):

**Уровень 1 — атомарная сигнатура (встроенные правила Wazuh):**
- Правило **31103** ("SQL injection attempt", уровень 7) матчит известные SQLi-паттерны (`union+`, `select+`, `null,null` и т.д.) в декодированном поле URL из Apache-лога.
- Правило **31106** ("A web attack returned code 200 (success)", уровень 6) — дочернее от 31103/31104/31105, срабатывает, когда тот же запрос также вернул HTTP 200 — то есть атака была *успешной*, а не просто предпринятой.
- MITRE ATT&CK: **T1190** (Exploit Public-Facing Application).

**Уровень 2 — кастомное корреляционное правило (local_rules.xml):**
```xml
<group name="sqli,web,attack,">
  <rule id="100101" level="12" frequency="5" timeframe="120">
    <if_matched_sid>31106</if_matched_sid>
    <same_source_ip />
    <description>Multiple SQL injection attempts from the same source - possible enumeration</description>
    <mitre>
      <id>T1190</id>
      <id>T1590</id>
    </mitre>
  </rule>
</group>
```
Правило эскалирует до уровня 12, когда правило 31106 срабатывает 5 и более раз от одного исходного IP в течение 120 секунд — фиксируя именно поведение enumeration, а не единичную попытку инъекции.

MITRE ATT&CK: **T1190** (Exploit Public-Facing Application) для начального вектора доступа, **T1590** (Gather Victim Network Information) для фазы enumeration/разведки.

**Инженерная заметка:** изначально было написано атомарное правило (100100) для прямого матчинга `UNION SELECT` через кастомный regex по полю `<url>`. Оно не срабатывало стабильно в реальных условиях, несмотря на успешную проверку синтаксиса — вероятно, из-за взаимодействия со встроенной цепочкой веб-правил Wazuh, которая уже резолвит событие через 31103/31106 до того, как кастомное параллельное правило может быть выбрано как "финальный" alert. В итоге корреляционное правило было перестроено на связку со встроенным 31106 вместо кастомного атомарного правила. Отдельное наблюдение при отладке: трекер частоты `if_matched_sid` в Wazuh, судя по всему, регистрирует только *самое глубокое* (наиболее специфичное) правило, сработавшее в цепочке декодирования, а не каждого промежуточного родителя — поэтому корреляция по 31103 (непосредственной SQLi-сигнатуре) не накапливала частоту корректно, пока параллельно срабатывало более специфичное дочернее правило 31106; переключение цели корреляции на 31106 решило проблему.

## 5. Обоснование классификации

Классифицировано как **SQL-инъекция — фаза Enumeration (высокая критичность)**. Обоснование:
- Структура payload-ов (UNION-based, с прогрессией от определения количества колонок до перечисления `information_schema`) — классическая последовательность ручной SQLi-разведки, а не случайный/безобидный трафик.
- Все запросы вернули HTTP 200 с телом ответа возрастающего размера (1843 → 2010 → 2591 → 6239 → 1850 байт), что соответствует возврату приложением структурированных данных (имена таблиц/колонок) атакующему.
- Запросы были направлены на `information_schema`, что указывает на целенаправленное картирование схемы БД перед прицельной попыткой извлечения данных (например, таблицы `users`).

---

## 6. Обоснование эскалации

Эскалировано до уровня 12 (высокий), а не оставлено на уровне атомарной сигнатуры (6-7), потому что:
- Корреляционное правило подтверждает *устойчивое, автоматизированного типа* поведение от одного источника, а не разовый зондирующий запрос — это существенно опаснее и указывает на активную, продолжающуюся атаку, а не фоновый шум сканеров.
- К моменту срабатывания alert'а последовательность enumeration уже дошла до `information_schema.columns`, то есть атакующий находился в одном шаге от полноценного запроса на извлечение учётных данных — эскалация на этом этапе позволяет вмешаться до фактической эксфильтрации данных (например, хешей паролей из таблицы `users`).

---

## 7. Меры реагирования

- **Немедленно:** заблокировать/ограничить по частоте исходный IP 192.168.1.133 на уровне межсетевого экрана или WAF.
- **На уровне приложения:** заменить конкатенацию сырых SQL-строк (как в DVWA) на параметризованные запросы / prepared statements в реальном production-коде (DVWA намеренно уязвима — в реальном развёртывании это основное исправление).
- **Эшелонированная защита:** развернуть набор правил WAF (например, OWASP CRS) перед приложением для перехвата UNION-based инъекций до того, как они достигнут уровня БД.
- **Мониторинг:** донастроить `frequency`/`timeframe` корреляционного правила по результатам дальнейшего тестирования, чтобы снизить риск пропуска более медленных, уклоняющихся от детекта попыток enumeration (например, если атакующий растягивает запросы за пределы окна в 120 секунд).
